## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| **서비스명** | PartyDiet (파티다이어트) |
| **슬로건** | 함께 짜고, 나누고, 보상받는 식단 |
| **핵심 가치** | 파티 내 식단 공유 + 파티 간 식단 거래 + 인센티브 기반 커뮤니티 |
| **타겟 사용자** | 다이어트 크루, 밀키트 애호가, 헬스 트레이너 & 트레이니 |
| **플랫폼** | iOS / Android / Web (멀티플랫폼) |

### 문제 정의

- 다이어트는 혼자 하면 지속성이 낮고, 식단 구성의 반복 고민이 많다.
- 잘 짜인 식단이 있어도 공유·유통 구조가 없어 자산화되지 못하고 소멸된다.
- 기존 앱은 개인 기록 중심이라, 그룹 협업과 식단 경제 생태계가 없다.

| 앱 | 주요 특징 | 약점 |
|----|-----------|------|
| **Mealime** | 700만+ 유저, 식재료 → 장보기 연동 | 그룹/커뮤니티 없음 |

| **Plan to Eat** | 가족 공유, 식단 플래너 | 그룹 협업 투표 없음 |

### 2.3 시장 갭 (기회)

> 기존 앱은 **개인 기록** 또는 **단순 그룹 공유** 수준에 머무른다.  
> **"그룹이 함께 식단을 만들고 → 다른 그룹과 거래하며 → 보상을 얻는"** 생태계를 구현한 서비스는 현재 없다.

## 3. 차별화 포인트

| 구분 | 기존 앱 | PartyDiet |
|------|---------|-----------|
| 식단 작성 | 개인 or 관리자 지정 | 파티장 직접 + 파티원 투표 협업 |
| 그룹 범위 | 단일 그룹 내 공유 | 그룹 간 식단 마켓플레이스 |
| 인센티브 | 없음 or 뱃지 | 식단 복사 시 포인트 보상 |
| 커머스 연계 | 없음 or 배송 앱 연동 | 식단 재료 기반 밀키트 제휴 |

## 4. 기술 스택

| 영역 | 기술 | 선택 이유 |
|------|------|-----------|
| **Frontend** | React 18+ (TypeScript) | 팀 역량, 생태계 |
| **Cross-platform** | Capacitor | iOS/Android/Web 단일 코드베이스, 비용 절감 |
| **상태관리** | Jotai | 가볍고 보일러플레이트 최소화 |
| **스타일링** | Tailwind CSS | 빠른 UI 구현 |
| **빌드** | Vite | 빠른 HMR |
| **Backend** | Firebase (Auth, Firestore, Functions) | 개인 개발자 적합 - 서버리스, 무료 티어 |
| **광고** | AdMob | 모바일 네이티브 광고 수익화 |

|--------|-----------|

| Cloud Functions | 2M 호출/월 |

## 5. 핵심 기능 상세

### 5.1 파티(그룹) 관리

#### 파티 생성 & 초대
- 사용자는 파티(그룹)를 생성하고 **고유 참여 코드** 또는 **Deep Link**를 통해 멤버 초대
- 인앱 초대장 수신 → 수락 시 자동 파티 가입
- Jotai `partyAtom`으로 실시간 멤버 리스트 및 권한(파티장/파티원) 관리

#### 파티 상태
```
생성 중 → 활성 → 종료(완료) → 공개 전환(선택)
```

---

### 5.2 식단 작성

#### 방식 1 - 파티장 직접 작성
- 파티장이 메뉴를 직접 구성하여 파티원에게 배포
- 파티원은 열람 및 활용만 가능

#### 방식 2 - 협업 식단 작성
- 파티원 누구나 메뉴 **제안** 가능
- 제안된 메뉴에 대해 파티원 **투표** 진행
- 투표 결과 상위 메뉴가 최종 식단으로 확정
- Firestore 실시간 리스너로 투표 현황 즉시 반영

#### 식단 구성 요소
- 메뉴명, 재료 목록, 칼로리/영양소 (탄단지)
- 이미지 첨부 (선택)
- 예산 정보 (1인분 기준 가격)

---

### 5.3 식단 마켓플레이스 (파티 간 교류)

#### 공개 전환
- 종료된 파티의 식단을 파티장이 **공개** 상태로 전환
- 공개된 식단은 마켓플레이스에 노출

#### 식단 검색 & 열람
- 검색 필터: 칼로리 범위, 예산, 기간, 태그 (키토, 벌크업, 저탄수 등)
- 타 파티 식단 상세 보기 → 네이티브 광고 노출

#### 식단 Import (복사)
- 타 파티 식단을 내 파티에 복사
- 복사 전 **보상형 동영상 광고 (AdMob Rewarded)** 시청 유도
- 복사 완료 시 원본 파티에 **포인트 자동 지급**

#### 비교 분석
- 타 파티의 **예산 대비 식단 구성비** 데이터 시각화 제공
- 시각화 라이브러리: Recharts 또는 Chart.js (번들 경량화 고려)

---

### 5.4 알림 & 동기부여

- 파티원 투표 참여 Push 알림 (Firebase Cloud Messaging)
- 식단 복사 시 포인트 획득 알림
- 식단 마감 D-day 알림

## 6. 수익 모델

### 6.1 AdMob 광고

| 광고 유형 | 노출 시점 | 예상 eCPM |
|-----------|-----------|-----------|
| 네이티브 광고 | 식단 검색 결과, 타 파티 식단 상세 | ₩300~₩800 |
| 보상형 동영상 | 식단 Import 전 | ₩1,500~₩3,000 |
| 배너 광고 | 앱 하단 (선택적) | ₩100~₩200 |

> 💡 개인 개발자 관점: 보상형 동영상이 단가 최고, 핵심 수익원으로 집중 설계 권장.

### 6.2 커머스 제휴 (애플리케이션 심화 단계)

- 식단 재료 기반 **밀키트 추천** → 쿠팡파트너스, 오아시스 제휴 딥링크 연결
- 구매 전환 시 **제휴 수수료** 수익 (쿠팡파트너스: 평균 3~5%)
- 구현 방법: 재료 태그 → 제휴 상품 매핑 테이블 (Firestore)

### 6.3 미래 수익화 옵션 (v2+)

| 옵션 | 설명 |
|------|------|
| 프리미엄 구독 | 광고 제거, 파티 최대 인원 확장, 식단 통계 고급 분석 |
| 트레이너 플랜 | 트레이너 → 다수 트레이니 파티 관리 도구 |

## 7. 포인트 활용 제안

> 파티 포인트는 파티장이 활용 가능. 아래 방안 중 단계적 도입 권장.

### Phase 1 (MVP 단계) - 구현 비용 최소
| 활용 방안 | 설명 |
|-----------|------|
| **식단 Import 횟수 확장** | 포인트 소모 시 광고 없이 Import 가능 (광고 스킵권) |
| **마켓 노출 우선순위 부스트** | 포인트 사용 시 내 식단이 검색 상위 노출 |

### Phase 2 (성장 단계)
| 활용 방안 | 설명 |
|-----------|------|
| **기프티콘 교환** | 파트너사 제휴 쿠폰 (GS25, 스타벅스 등) |
| **밀키트 할인 쿠폰** | 제휴 커머스 할인 코드 교환 |
| **프리미엄 기능 이용권** | 고급 영양소 분석, 식단 PDF 다운로드 등 |

### Phase 3 (커뮤니티 성숙 단계)
| 활용 방안 | 설명 |
|-----------|------|
| **트레이너 인증 뱃지** | 포인트 누적량 기반 신뢰 지표 |
| **파티 홍보 광고** | 포인트로 내 파티/식단 앱 내 배너 노출 |

## 9. 제약 사항 및 리스크

### 9.1 개인 개발자 제약

| 리스크 | 대응 방안 |
|--------|-----------|
| 개발 공수 부족 | MVP 기능을 최소화, Phase 1에 집중 |
| 서버 비용 | Firebase 무료 티어 최대 활용, Firestore 쿼리 최적화 |
| 마케팅 예산 없음 | 커뮤니티(다이어트 카페, 인스타) 바이럴, 초대 코드 공유 유도 |
| 앱스토어 심사 | AdMob 정책, 개인정보처리방침 사전 준비 |

### 9.2 기술적 리스크

| 리스크 | 대응 방안 |
|--------|-----------|
| Firestore 실시간 리스너 비용 | 투표는 단기 이벤트이므로 Cloud Functions로 집계 후 스냅샷 저장 |
| Capacitor 플러그인 호환성 | AdMob 플러그인(@capacitor-community/admob) 버전 고정 |
| 포인트 시스템 어뷰징 | 식단 복사 1회당 1포인트 상한, 동일 IP 중복 방지 로직 |

### 9.3 스코프 아웃 (v1 제외)

- 칼로리 자동 계산 / 영양소 DB (외부 API 비용)
- AI 식단 추천
- 실시간 채팅
- 결제 기반 프리미엄 구독

## 부록 - Firestore 주요 컬렉션 설계 (초안)

```
users/{uid}
  - displayName, avatarUrl, partyIds[], totalPoints

parties/{partyId}
  - name, code, leaderId, memberIds[], status(active|closed|public)
  - createdAt, closedAt

parties/{partyId}/meals/{mealId}
  - title, items[], totalCalorie, budget, imageUrl
  - status(draft|voted|final)
  - voteCount, importCount

marketplace/{mealId}  ← 공개된 식단의 denormalized 사본
  - partyId, title, tags[], totalCalorie, budget
  - importCount, createdAt

pointLogs/{logId}
  - uid, partyId, type(earn|spend), amount, reason, createdAt
```
