# Harness Engineering Principles v2.0

이 문서는 하네스 엔지니어링 구축 프로젝트의 공통 실행 원칙이다.  
모든 에이전트는 아래 원칙을 우선 적용한다.

> **원칙 간 충돌 시 우선순위: 보안 > 완료 기준 > 성능**

---

## 1) 행동 제약 (Guardrails)

- 허용된 출력 형식(JSON / 코드 / 로그 형식)을 반드시 지킨다.
- 요구되지 않은 외부 API 호출, 임의 파일 수정, 위험한 명령 제안을 금지한다.
- 추측이 필요할 때는 보수적으로 선택하고, 불확실성을 명시한다.
- 실패 시 원인과 수정 방향을 구조적으로 반환한다.
- **[신규]** 환경 변수 및 시크릿 키는 출력/로그에 포함하지 않는다.  
  디버그 로그, 오류 메시지, 반환값 어디에도 키 값이 노출되어선 안 된다.

---

## 2) 실행 환경 (Environment)

- 각 에이전트는 자신의 책임 범위 안에서만 동작한다.
- 개발 산출물은 지정된 경로(`artifacts/` 등)에만 작성한다.
- 입력 데이터(계획, 디자인, 테스트 로그)를 우선 사용하고, 임의 확장을 최소화한다.
- 환경 변수/키가 없으면 우회하지 말고 명확한 오류를 반환한다.
- **[신규]** 모든 실행 입력/출력/결정 근거를 구조화된 로그로 남긴다.  
  로그는 `logs/` 경로에 JSON Lines 형식으로 저장하고, 시크릿 값은 마스킹 처리한다.

---

## 3) 에이전트 협업 (Agent Collaboration) `[신규]`

- 에이전트 간 작업 위임은 명시적 인터페이스(입력 스키마 + 완료 조건)를 통해서만 수행한다.
- 위임 시 컨텍스트(목표, 제약, 현재 상태)를 함께 전달한다. 암묵적 공유 상태에 의존하지 않는다.
- 에이전트 간 결과 충돌 시, 오케스트레이터가 우선권을 갖는다. 에이전트 단독으로 충돌을 해소하지 않는다.

---

## 4) 검증 루프 (Verification Loop)

- 결과 생성 후 즉시 테스트/검증을 수행한다.
- 실패 시: 로그 분석 → 원인 분류 → 수정안 생성 → 재시도.  
  **[보완]** 기본 최대 재시도 횟수는 **3회**이며, 에이전트별 스펙에서 오버라이드 가능하다.
- 같은 실패가 2회 이상 반복되면 전략을 교체한다. 동일 접근을 반복하지 않는다.
- **[보완]** 재시도 한도 초과 시, 오케스트레이터에 실패 컨텍스트(시도 횟수, 마지막 오류, 시도한 전략 목록)를 반환한다.

---

## 5) 완료 조건 (Definition of Done)

완료 상태는 세 단계로 구분한다.

| 상태 | 조건 | 처리 |
|------|------|------|
| **완료** | 모든 최소 기준 충족 | 정상 종료 |
| **부분 완료** | 기준 일부 충족 | 충족/미충족 항목 명시 후 상위 전달 |
| **실패** | 재시도 한도 초과 | 실패 컨텍스트 반환 |

최소 기준:
- 필수 기능 충족
- 테스트 통과
- 형식/스키마 요구사항 충족

**[보완]** 부분 완료는 미완료가 아니다. 충족한 항목과 미충족 항목을 명시하여 상위 오케스트레이터에 전달한다.

---

## 핵심 패러다임

> 좋은 프롬프트보다, 검증 가능한 기준이 더 중요하다.
